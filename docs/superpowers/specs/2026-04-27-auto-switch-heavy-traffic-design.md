# 自动切换大流量策略组设计

## 背景

当前 `traffic-monitor` 已经能持续读取 Mihomo `/connections`，并按分钟聚合域名、设备、出口节点等维度的流量数据。现阶段项目偏向“发现问题”，还不能在发现某个域名出现异常大流量时，自动执行策略切换。

本次扩展的目标不是接管 Mihomo 的规则判断，也不是维护域名白名单，而是在保留现有规则体系的前提下，为用户提供一个自动执行预案的能力：

- 当任意域名在 1 分钟窗口内的累计流量超过用户配置阈值时，触发一次自动切换
- 自动切换只负责把用户选中的策略组切换到用户预设的目标节点
- 规则归属、域名匹配、流量应命中哪个策略组，仍由用户现有的 Mihomo 规则负责

## 非目标

本次设计明确不包含以下内容：

- 不维护域名白名单、黑名单或域名到策略组的映射表
- 不尝试根据连接链路自动推断“应该切哪个策略组”
- 不展示或控制 `url-test`、`load-balance`、`relay` 等不适合作为手动目标的策略组类型
- 第一版不做自动回切，不在流量回落后自动恢复原节点
- 不修改现有流量统计口径和钻取查询行为

## 用户体验目标

用户只需要完成一次简单配置：

1. 启用自动切换
2. 设置“1 分钟累计流量阈值”
3. 设置“触发冷却时间”
4. 从 Mihomo 读取可控策略组，仅展示 `select` 和 `fallback`
5. 为每个启用的策略组选择一个目标节点

之后系统自动完成：

- 监测任意域名的分钟累计流量
- 超阈值时执行一次预设切换
- 记录触发日志，便于回溯

## 方案比较

### 方案 A：域名规则映射

为每个域名单独维护 `域名 -> 策略组 -> 节点` 规则。

优点：

- 精准可控

缺点：

- 维护成本高
- 与用户当前“依赖 Mihomo 规则分流”的方式重复

### 方案 B：统一触发，分组独立目标节点

只判断“任意域名在 1 分钟内是否超过阈值”，一旦触发，就把用户启用的策略组分别切到各自预设的目标节点。

优点：

- 配置简单
- 与现有规则体系解耦
- 便于在复杂策略组场景下稳定落地

缺点：

- 触发不区分具体域名业务归属

### 方案 C：自动推断归属策略组

系统根据 `chains` 或当前策略组结构自动猜测该切哪个组。

优点：

- 理论上配置最少

缺点：

- 复杂策略组嵌套下容易误判
- 可解释性差，排障困难

### 结论

采用方案 B：统一触发，分组独立目标节点。

## 功能设计

### 1. 自动切换配置

新增自动切换配置模块，持久化到 SQLite。配置包含：

- `enabled`：总开关
- `threshold_bytes_per_minute`：1 分钟累计流量阈值
- `cooldown_seconds`：冷却时间
- `group_targets`：策略组目标列表

其中 `group_targets` 的每一项包含：

- `group_name`：策略组名称
- `target_proxy`：目标节点名称
- `enabled`：该策略组是否参与自动切换

### 2. 可控策略组来源

系统从 Mihomo Controller 拉取策略组元数据，仅展示：

- `type = select`
- `type = fallback`

明确排除：

- `url-test`
- `load-balance`
- `relay`
- 其他不支持用户直接固定目标的类型

每个策略组的目标节点列表仅来自该组当前可选项，不能自由输入。

### 3. 触发条件

触发逻辑统一且全局生效：

- 监控仍基于当前 `/connections` 的增量采集结果
- 以 `host` 为键统计当前分钟窗口的累计流量
- 累计值 = `upload + download`
- 当任意 `host` 在该分钟窗口内超过阈值时，进入自动切换流程

不需要用户维护域名列表，也不基于域名做额外匹配。

### 4. 执行动作

触发后系统按配置逐个处理启用的策略组：

1. 检查当前是否在冷却期内
2. 遍历所有已启用的 `group_targets`
3. 读取当前策略组实际选中项
4. 如果当前已经是 `target_proxy`，跳过
5. 如果不同，则调用 Mihomo API 将该策略组切到目标节点
6. 记录成功或失败结果

所有策略组共享同一套触发条件，但每个策略组使用各自独立的目标节点。

### 5. 冷却与去抖

为避免频繁切换，第一版实现以下安全边界：

- 触发后进入冷却期，默认值可配置
- 冷却期内再次发现任意域名超阈值，也不重复执行切换
- 同一分钟窗口内只允许触发一次
- 如果策略组已经处于目标节点，不重复调用切换接口
- 第一版不做自动回切

### 6. 触发日志

系统保存最近触发记录，至少包含：

- `triggered_at`
- `host`
- `total_bytes`
- `window_start`
- `window_end`
- 参与执行的策略组结果
- 错误信息（如有）

日志用于页面展示与问题排查，不参与统计聚合。

## 后端设计

### 数据库

新增表建议如下：

#### `auto_switch_settings`

- `key TEXT PRIMARY KEY`
- `value TEXT NOT NULL`

用于保存总开关、阈值、冷却时间等全局配置，复用当前 `app_settings` 风格也可以。若希望减少表数量，也可以直接继续写入 `app_settings`。

#### `auto_switch_group_targets`

- `group_name TEXT PRIMARY KEY`
- `target_proxy TEXT NOT NULL`
- `enabled INTEGER NOT NULL`

保存每个策略组的目标节点配置。

#### `auto_switch_events`

- `id INTEGER PRIMARY KEY AUTOINCREMENT`
- `triggered_at INTEGER NOT NULL`
- `host TEXT NOT NULL`
- `total_bytes INTEGER NOT NULL`
- `window_start INTEGER NOT NULL`
- `window_end INTEGER NOT NULL`
- `results_json TEXT NOT NULL`
- `error TEXT NOT NULL DEFAULT ''`

用于记录触发历史。

### 内存状态

在 `service` 中新增运行时状态：

- 当前分钟的 `host -> bytes` 累计窗口
- 最近一次触发时间
- 最近一次触发窗口标识

建议结构：

- `hostMinuteWindow map[string]*hostTrafficWindow`
- `autoSwitchState`，包含冷却与最近触发信息

`hostTrafficWindow` 至少包含：

- `bucket_start`
- `bucket_end`
- `total_bytes`

### 采集链路接入点

逻辑接入当前 `processConnections` 流程：

1. 处理增量连接流量
2. 生成 `trafficLog`
3. 写入聚合缓冲
4. 额外把本批增量按 `host` 更新到分钟窗口
5. 检查是否有 `host` 超阈值
6. 如果满足条件，则尝试执行自动切换

这样可以最大化复用现有采集逻辑，不需要额外调度器。

### Mihomo API 交互

需要新增两类 Controller 调用：

- 查询代理组详情，用于拿到所有 `select`/`fallback` 策略组及其候选项、当前选中项
- 执行策略组切换，用于把指定组切到目标节点

后端需要对返回结果做校验：

- 策略组是否存在
- 组类型是否允许控制
- `target_proxy` 是否是该组的合法候选项

## 前端设计

页面新增一个“自动切换”配置区，建议放在设置相关区域附近，但与 Mihomo 基础连接配置分开。

### 配置区内容

- 自动切换总开关
- 1 分钟阈值输入框
- 冷却时间输入框
- 刷新策略组按钮
- 策略组配置表格

策略组表格每行包括：

- 启用复选框
- 策略组名称
- 策略组类型
- 当前选中节点
- 目标节点下拉框

目标节点下拉框的选项来自对应策略组的候选项。

### 触发记录区

展示最近若干次自动切换事件：

- 触发时间
- 命中的域名
- 分钟累计流量
- 执行结果摘要

## 接口设计

建议新增以下接口：

- `GET /api/auto-switch/settings`
- `PUT /api/auto-switch/settings`
- `GET /api/auto-switch/groups`
- `GET /api/auto-switch/events`
- `POST /api/auto-switch/test`

说明：

- `settings` 负责读写总开关、阈值、冷却时间和策略组目标配置
- `groups` 从 Mihomo 拉取当前可控策略组与候选节点
- `events` 返回最近触发记录
- `test` 用于手动测试切换逻辑，便于验证配置是否正确

`test` 可以是第一版可选项。如果控制工作量，也可以先不做。

## 错误处理

需要覆盖以下场景：

- Mihomo 未配置或连接失败
- 拉取策略组列表失败
- 策略组不存在或类型不支持
- 目标节点不在候选项中
- 切换接口调用失败
- 数据库存储失败

设计原则：

- 自动切换失败不能影响主采集流程
- 所有失败都需要记录日志与事件结果
- 页面要能看到失败原因

## 测试策略

至少覆盖以下测试：

- 分钟窗口内的 `host` 累计逻辑正确
- 跨分钟后窗口正确轮转
- 超阈值后只触发一次
- 冷却期内不会重复触发
- 已在目标节点时不会重复切换
- 仅返回 `select` 和 `fallback` 组
- 目标节点校验正确
- Mihomo 接口失败不会中断采集流程

## 风险与后续演进

### 已知风险

- 因为不区分域名归属策略组，所以某个域名超阈值会触发所有启用的策略组一起执行预案
- 第一版不自动回切，需要用户自行恢复
- 当前项目采样粒度由 `/connections` 轮询频率决定，极短时高峰可能存在统计偏差

### 后续可演进方向

- 增加“仅切指定策略组子集”的多套预案
- 增加按域名后缀、按规则集的高级匹配
- 增加自动回切策略
- 增加手动试运行与最近成功节点记忆

## 最终结论

第一版聚焦实现一个简单、稳定、可控的自动化执行器：

- 不碰用户规则判断逻辑
- 不维护域名映射
- 只根据“任意域名 1 分钟流量超阈值”统一触发
- 只控制用户选定的 `select` / `fallback` 策略组
- 每个策略组切到用户预设的目标节点

这样既能满足“大流量出现时快速切换预案”的需求，也能保持当前项目轻量、直接、易于定制的定位。
